## 当前架构定位

本项目当前版本采用“数据库中心”架构。

- MySQL 是核心业务数据源与状态落点。
- Qt Widgets 客户端通过 Service 和 DAO 直接完成主要业务读写。
- HTTP 仅保留用于认证相关能力，包括登录与注册。
- TCP 长连接、消息推送、协议分发不属于当前版本主链路，相关设计已停止作为核心实现方向。

## 当前分层约定

### 1. UI 层

位于 ui/ 和根目录窗口文件中，负责：

- 页面展示
- 用户交互
- 调用 Service 获取或提交业务数据
- 在页面切换时触发必要刷新

UI 层不负责：

- 直接拼装 SQL
- 承担数据库事务
- 依赖 TCP 协议消息驱动页面状态

### 2. Service 层

位于 services/，负责：

- 业务规则编排
- 多个 DAO 之间的协作
- 操作日志写入
- 页面需要的聚合数据组织

Service 层不负责：

- 内置演示数据兜底
- 在正常读取路径中自动补默认数据
- 假定存在 TCP 指令下发或推送确认

### 3. DAO 层

位于 database/dao/，负责：

- 单表或明确事务范围内的数据访问
- SQL 执行与结果映射
- 返回明确的错误信息

DAO 层不负责：

- 页面展示逻辑
- 业务文案拼装
- UI 刷新控制

### 4. 认证链路

认证属于例外路径：

- 登录使用 HTTP 接口
- 注册使用 HTTP 接口
- 登录成功后再进入数据库主业务流程

因此项目当前是“数据库中心业务 + HTTP 认证入口”的混合架构，而不是纯前后端 TCP 通信架构。

## 当前版本明确不做的事

- 不将 TCP 作为当前主业务通信通道
- 不在设备控制中保留伪协议报文下发链路
- 不在业务查询时自动灌入默认设备或默认分类
- 不使用本地模拟设备数据替代数据库结果

## 后续重构原则

### 1. 数据一致性优先

页面状态应以数据库真实结果为准，页面切换后要主动刷新关键模块，避免预加载页面出现状态陈旧。

### 2. 允许场景部分成功

场景执行中若个别设备失败或离线，其余设备仍继续执行。
执行结果需要记录成功与失败明细，并在 UI 上体现部分成功状态。

### 3. 防止 UI 假死

历史查询、环境读取、批量写日志、备份等耗时数据库操作，后续需要逐步改为异步执行。

### 4. 保持边界清晰

任何新功能都优先沿用 UI -> Service -> DAO -> DatabaseManager 的主路径，不再新增伪 TCP 协议层。
